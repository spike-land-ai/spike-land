## The Hook

Most legacy Next.js apps are not hard to migrate because they are old.

They are hard to migrate because nobody can still see the structure.

Years of framework conventions, Vercel-specific assumptions, mixed rendering models, middleware glue, half-finished server abstractions, and business logic leaking through UI code make the repo feel bigger than it is. Teams look at that mess and conclude the same thing:

"We probably need a rewrite."

Usually, they do not.

They need a way to read the architecture first.

That is the whole idea behind this service.

We do not start by rewriting files.  
We start by extracting the actual structure of the system: routing patterns, dependency boundaries, runtime assumptions, edge concerns, data-loading behavior, and framework-specific coupling.

Once that structure is visible, the migration stops being a heroic rewrite and becomes what it should have been all along:

a sequence of explicit, reviewable transformations.

## The Method Behind The Migration

The migration work sits on the same foundation as [BAZDMEG](https://spike.land/bazdmeg):

- requirement clarity before code generation
- safe entrypoints before destructive actions
- repo state read before mutation
- deterministic passes before agent escalation
- tests as truth, not vibes
- ownership of every diff

That matters because migrations are risky work. They touch routing, runtime, deployment, caching, environment handling, server assumptions, asset behavior, and business logic boundaries all at once.

If you start with an invasive pipeline, you are gambling.

If you start with structure, you can make the migration auditable.

That is why the flow is:

1. read the repo
2. detect the architecture
3. preview the transformation
4. apply the mechanical pass
5. verify the output
6. escalate only the edge cases that deserve human judgment

The same principle shows up elsewhere in our work too. When we reorganized 990 TypeScript files in our monorepo, the trick was not taste. It was dependency analysis. The architecture was already in the imports. The script just read it.

Migration works the same way.

The codebase already contains the truth about what it is. You just need a system that can read it.

## What We Are Actually Migrating To

This is not a generic "leave Next.js somehow" service.

It is a migration from Next.js toward an explicit, edge-native stack:

| From | To |
|---|---|
| Next.js App Router / Pages Router | TanStack Router |
| `next.config.js` | `vite.config.ts` |
| Vercel deployment model | Cloudflare Workers |
| Next.js middleware | Hono middleware |
| `getServerSideProps`, ISR patterns | explicit loaders, client-first rendering, edge handlers |
| `next/image`, `next/font` | direct imports and standard platform-native handling |

That does not mean every feature maps 1:1.

It means the target stack is opinionated, explicit, and cheaper to reason about.

That is the real benefit. Not "new shiny tool."  
A simpler runtime model, faster local loops, fewer hidden conventions, and less framework rent.

## Why Sophisticated Legacy Projects Are Still In Scope

Because the migration is not treated as a giant rewrite.

This is where the technical background matters.

Hard codebases usually break migration efforts for three reasons:

1. **Framework coupling is implicit.**  
   Data loading, auth, routing, rendering, and deploy assumptions are mixed together.

2. **Business logic is in the wrong layer.**  
   Server logic leaks into UI, UI state leaks into backend assumptions, and nobody is sure what is authoritative.

3. **The team cannot separate mechanical work from judgment.**  
   So everything gets treated like custom engineering, which makes the project too expensive to attempt.

We handle those problems differently.

Mechanical changes get formalized.

Examples:
- route file rewrites
- config conversion
- import replacement
- directory reshaping
- environment variable remapping
- deploy config changes
- asset and middleware rewiring

Judgment-heavy changes get isolated.

Examples:
- Vercel-only platform assumptions
- long-lived Node.js runtime dependencies
- custom SSR behavior that is really business logic
- ad hoc middleware side effects
- code that relied on undocumented Next.js behavior

That split is what makes advanced repos workable.

Not because the hard parts disappear, but because they stop contaminating the whole migration.

## What The Technical Pipeline Looks Like

This is the simplified version of how the migration is approached.

### 1. Safe Read

No destructive pipeline first.

We inspect the repo state, understand the routing model, identify framework-specific imports, detect runtime dependencies, and classify the codebase by concern.

This is the migration equivalent of `bazdmeg:status`.

### 2. Preview

Before applying anything, we generate a preview of what the migration would do.

What will move.
What will be rewritten.
What will be left alone.
What looks suspicious.
What likely needs human review.

This is the migration equivalent of `bazdmeg:dry-run`.

### 3. Deterministic Pass

Only after the preview matches intent do we run the real transformation.

This includes:
- routing translation
- config rewrites
- import replacement
- edge/runtime adaptation
- deploy target changes
- project structure cleanup

The key is that the boring parts are explicit and repeatable.

### 4. Verification

After the transformation:
- TypeScript must pass
- lint must pass
- relevant tests must pass
- business-critical flows must still hold
- generated diff must be understandable by a human reviewer

If the diff cannot be explained, it is not done.

### 5. Escalation

Only the real edge cases get human attention.

That is where support earns its value.

## What Does Not Migrate Automatically

This is not magic, so the boundaries matter.

The following usually need custom treatment:

- deeply Vercel-specific infrastructure behavior
- code that assumes long-lived Node.js APIs or server processes
- undocumented Next.js internals
- application logic that was accidentally encoded inside framework behavior
- repos with weak test coverage on core business paths

We say this up front because false confidence is expensive.

A migration service should reduce risk, not hide it.

## FAQ

**Is this a rewrite?**  
No. The whole point is to avoid treating it like one. The migration starts with extracting structure, then applying explicit transformations.

**Can we roll back?**  
Yes. Tier 2 returns a diff. Tier 3 opens a PR. Your main branch does not change until you merge.

**Does this work for App Router and Pages Router?**  
Yes. Both are in scope.

**What if the app has blockers?**  
We tell you before paid migration work starts. If a repo contains real blockers, the correct answer is not to bluff.

**Why not just use an AI coding agent directly?**  
Because a raw agent without structure will confidently produce migration slop. The value here is not "AI exists." The value is requirement discipline, structural reading, deterministic transformation, and human review only where needed.

**What is the link to BAZDMEG?**  
Same mindset: safe read first, preview second, destructive action last, ownership always. The method matters more than the model.
